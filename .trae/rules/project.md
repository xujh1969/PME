# PME编写Markdown编辑器 — AI开发规范（TRAE Rules）

## 一、Karpathy四大原则（最高优先级，每次开发前强制执行）

### 1. Think Before Coding — 编码前思考
- 输出代码前，先显式列出本次开发的**全部前置假设**，不隐藏任何隐性决策
- 需求存在多种解读时，逐条列出所有方案并询问用户确认，禁止自行猜测实现
- 遇到架构、表关联、接口字段、权限逻辑等模糊点，立刻暂停并提问，不擅自补全
- 主动对比多种实现方案，优先推荐最简可行方案
- 需求超出设计文档范围或存在冲突时，直接推回需求并说明原因

### 2. Simplicity First — 简洁第一
- 禁止开发需求未提及的拓展功能、预留配置、通用抽象
- 代码极致精简：同等功能优先短实现，冗余超4倍必须重构
- 不额外增加未要求的缓存、日志、权限校验、异常捕获逻辑
- 统一复用项目现有封装（ApiResult、PageRequestDto、jwtAuthGuard），不重复造轮子

### 3. Surgical Changes — 手术式修改
- 仅修改用户明确指定的文件/代码块，无关代码、格式、缩进、注释一律不动
- 无重构指令时，禁止优化旧代码、统一全局格式
- 每一行修改必须可追溯至用户需求，无无关附带改动

### 4. Goal-Driven Execution — 目标驱动执行
- 接收需求后，先输出**可验证成功标准清单**，确认目标后再编码
- 功能实现类：拆分校验点（接口返回格式、数据库约束、前端渲染、权限拦截、分页逻辑）
- Bug修复类：先复现逻辑，再修复，最后验证
- 所有代码完成后自动对照成功标准自查，不满足则迭代修正

## 二、AI开发输出标准流程（固定顺序，不可跳过）

1. **Karpathy原则前置校验**：梳理需求假设、列出多方案、标记模糊点提问
2. **输出目标验证清单**：明确本次开发完成后的校验标准
3. **全局规则匹配**：确认分层、复用规范、行数限制、数据库性能要求
4. **代码分层设计**：规划新增/修改文件列表
5. **编写代码**：严格遵守简洁第一、手术式修改
6. **输出交付物**：新增文件完整代码、修改文件差异代码、功能验证说明
7. **自动测试**：在代码输出后，自动启动项目测试套件，验证功能是否正常
8. **代码质量检查**：使用ESLint、Prettier等工具检查代码质量，确保符合规范
9. **用户反馈**：每次修改后，**启动自动测试**，确保功能正常
10.本程序因为主要在Tauri框架里运行，需要在代码修改好后保证在Tauri的测试环境以及编译为exe后各个功能都需要正确

### 环境一致性检查规则（必须执行）
- **开发环境测试**：使用 `npm run tauri dev` 启动，验证所有新增/修改的功能
- **生产环境测试**：使用 `npm run tauri build` 打包，安装并运行生成的 exe 安装包，验证以下关键功能：
  1. 配置界面的字体切换功能（中文字体、英文字体、代码字体）
  2. 主题切换功能（浅色/深色主题）
  3. 帮助菜单的使用说明功能
  4. 涉及动态样式注入的功能
- **差异排查**：如果开发环境正常但生产环境异常，优先检查以下原因：
  1. CSS 样式优先级问题（使用 `!important` 或内联样式）
  2. 动态样式注入的 CSP 策略限制
  3. 资源路径问题（开发路径 vs 打包后路径）
  4. 异步初始化顺序问题（确保配置在渲染前完成）
