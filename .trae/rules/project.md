# PME编写Markdown编辑器 — AI开发规范（TRAE Rules）

## 一、Karpathy四大原则（最高优先级，每次开发前强制执行）

### 1. Think Before Coding — 编码前思考
- 输出代码前，先显式列出本次开发的**全部前置假设**，不隐藏任何隐性决策
- 需求存在多种解读时，逐条列出所有方案并询问用户确认，禁止自行猜测实现
- 遇到架构、表关联、接口字段、权限逻辑等模糊点，立刻暂停并提问，不擅自补全
- 主动对比多种实现方案，优先推荐最简可行方案
- 需求超出设计文档范围或存在冲突时，直接推回需求并说明原因

### 2. Simplicity First — 简洁第一
- 禁止开发需求未提及的拓展功能、预留配置、通用抽象
- 代码极致精简：同等功能优先短实现，冗余超4倍必须重构
- 不额外增加未要求的缓存、日志、权限校验、异常捕获逻辑
- 统一复用项目现有封装（ApiResult、PageRequestDto、jwtAuthGuard），不重复造轮子

### 3. Surgical Changes — 手术式修改
- 仅修改用户明确指定的文件/代码块，无关代码、格式、缩进、注释一律不动
- 无重构指令时，禁止优化旧代码、统一全局格式
- 每一行修改必须可追溯至用户需求，无无关附带改动

### 4. Goal-Driven Execution — 目标驱动执行
- 接收需求后，先输出**可验证成功标准清单**，确认目标后再编码
- 功能实现类：拆分校验点（接口返回格式、数据库约束、前端渲染、权限拦截、分页逻辑）
- Bug修复类：先复现逻辑，再修复，最后验证
- 所有代码完成后自动对照成功标准自查，不满足则迭代修正

## 二、AI开发输出标准流程（固定顺序，不可跳过）

1. **Karpathy原则前置校验**：梳理需求假设、列出多方案、标记模糊点提问
2. **输出目标验证清单**：明确本次开发完成后的校验标准
3. **全局规则匹配**：确认分层、复用规范、行数限制、数据库性能要求
4. **代码分层设计**：规划新增/修改文件列表
5. **编写代码**：严格遵守简洁第一、手术式修改
6. **输出交付物**：新增文件完整代码、修改文件差异代码、功能验证说明
7. **自动测试**：在代码输出后，自动启动项目测试套件，验证功能是否正常
8. **代码质量检查**：使用ESLint、Prettier等工具检查代码质量，确保符合规范
9. **用户反馈**：每次修改后，**启动自动测试**，确保功能正常
10.本程序因为主要在Tauri框架里运行，需要在代码修改好后保证在Tauri的测试环境以及编译为exe后各个功能都需要正确

### 环境一致性检查规则（必须执行）
- **开发环境测试**：使用 `npm run tauri dev` 启动，验证所有新增/修改的功能
- **生产环境测试**：使用 `npm run tauri build` 打包，安装并运行生成的 exe 安装包，验证以下关键功能：
  1. 配置界面的字体切换功能（中文字体、英文字体、代码字体）
  2. 主题切换功能（浅色/深色主题）
  3. 帮助菜单的使用说明功能
  4. 涉及动态样式注入的功能
- **差异排查**：如果开发环境正常但生产环境异常，优先检查以下原因：
  1. CSS 样式优先级问题（使用 `!important` 或内联样式）
  2. 动态样式注入的 CSP 策略限制
  3. 资源路径问题（开发路径 vs 打包后路径）
  4. 异步初始化顺序问题（确保配置在渲染前完成）

## 三、CSS 与样式硬规则（防止样式污染与开发/生产不一致）

> 本章节为强制规范，每次涉及 UI/样式修改时必须遵守。

### 1. 禁止在 JS 中写 display 内联样式
- ❌ 禁止：`el.style.display = "none" / "block" / "flex" / "grid"`
- ✅ 统一使用 `classList` 切换工具类：
  - 隐藏：`el.classList.add("is-hidden")` 或 `el.classList.toggle("is-hidden", cond)`
  - 显示：`el.classList.remove("is-hidden")`
  - 判断隐藏态：`el.classList.contains("is-hidden")`
- ✅ 工具类定义在 `src/styles/utilities.css`：`.is-hidden { display: none !important; }`
- ✅ 如需切换不同布局（flex/grid），定义语义类如 `.as-grid { display: grid }`、`.as-flex { display: flex }`，通过 `classList` 切换

### 2. 禁止在 HTML 模板字符串中写 `style="display:..."`
- ❌ 禁止：`<div style="display: none;">...</div>`
- ✅ 改用：`<div class="... is-hidden">...</div>` 作为初始隐藏态

### 3. CSS 文件组织规范
- 所有样式必须放在 `src/styles/` 目录下对应模块文件中，不得新建独立 CSS 文件
- 单个 CSS 文件行数阈值：**1200 行**。超过时需按子模块拆分，并经用户确认
- 禁止将新样式堆到 `settings.css` 或 `ai-modal.css` 中（这两个文件已偏大）
- 新增样式优先归入最贴近的模块文件；如无合适模块，先与用户确认是否新建模块

### 4. `!important` 使用约束
- 仅在覆盖第三方库（如 ProseMirror、TipTap）默认样式时允许使用
- 业务样式禁止使用 `!important`，应通过提升选择器特异性解决优先级问题
- 新增 `!important` 时必须在代码中注释说明理由

### 5. 开发/生产一致性验证流程（每次 UI/样式修改后必跑）
按顺序执行三步对比：
```bash
npm run dev         # 开发环境（HMR），验证功能正常
npm run build       # 生产构建
npm run preview     # 在 http://127.0.0.1:4173 预览生产产物
```
对关键界面（AI 弹窗、设置面板、欢迎页、编辑器）做**视觉对比**，重点检查：
- 元素显示/隐藏是否正确
- flex/grid 布局是否生效
- 深色主题色值是否一致
- 动态注入样式是否生效

### 6. Tauri 双环境验证（按本规范第二章第10条）
- `npm run tauri dev` 验证开发态
- `npm run tauri build` 后安装 exe，验证生产态
- 重点测试：字体切换、主题切换、帮助菜单、动态样式注入功能

### 7. 提交前自查清单
- [ ] 是否新增了 `style.xxx =` 内联样式？（应为否）
- [ ] 是否新增了 `style="..."` 内联属性？（应为否，除非是 CSS 变量动态值如 `--font-family-xxx`）
- [ ] 是否新增了 `!important`？（需注释说明理由）
- [ ] 新增 CSS 是否放在对应模块文件中，而非堆到 settings.css / ai-modal.css？
- [ ] 是否执行了 `npm run preview` 视觉对比？
- [ ] 是否执行了 `npm run tauri dev` + `npm run tauri build` 双环境验证？
